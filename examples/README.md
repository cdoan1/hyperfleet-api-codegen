# Examples

This directory contains example types demonstrating the HyperFleet envelope pattern.

## Directory Structure

### `original/` - Simple Teaching Example

**File**: `sample_types.go`

A compact, easy-to-understand example showing the basic envelope pattern:

- **ClusterSpec** with 5 envelope fields
- **HostedClusterPassthrough** with nested types (Release, EtcdSpec)
- Demonstrates all marker types:
  - `+k8s:openapi-gen=false` for hidden fields
  - `+hyperfleet:write-mode=mutable|immutable|service-set`
  - `+openshift:enable:FeatureGate=HyperFleetEtcdConfig`

**Marker scanner results**: 15 total fields

```bash
./bin/marker-scanner -input-dirs=examples/original -verbose
```

**Use this when**:
- Learning the envelope pattern
- Testing marker scanner
- Quick demos and documentation

### `hypershift/` - Realistic Production-Like CRDs

**Files**: `cluster_types.go`, `nodepool_types.go`

Complete, production-ready CRD examples showing real HyperFleet → HyperShift integration:

#### Cluster
- Full Kubernetes resource structure (TypeMeta, ObjectMeta, Spec, Status)
- 7 envelope fields (displayName, deleteProtection, expirationTimestamp, properties, accountId, creatorARN, internalId)
- Embeds `HostedClusterSpecPassthrough` from generated code
- Includes ClusterStatus with conditions
- kubebuilder markers for CRD generation

#### NodePool
- Full Kubernetes resource structure
- 6 envelope fields (clusterRef, displayName, autoRepair, labels, accountId, internalPoolId)
- ClusterReference type for parent cluster
- Embeds `NodePoolSpecPassthrough` from generated code
- Includes NodePoolStatus with replica counts

**Marker scanner results**: 33 total fields

```bash
./bin/marker-scanner -input-dirs=examples/hypershift -verbose
```

**Use this when**:
- Understanding real-world CRD structure
- Seeing how envelope fields integrate with passthrough
- Planning production CRD implementation

## The Envelope Pattern

Both examples demonstrate the core pattern:

```go
type ClusterSpec struct {
    // === HyperFleet Envelope Fields ===
    // Hand-written fields with explicit markers
    
    // +hyperfleet:write-mode=mutable
    DisplayName string `json:"displayName,omitempty"`
    
    // +k8s:openapi-gen=false
    // +hyperfleet:write-mode=service-set
    AccountID string `json:"accountId,omitempty"`
    
    // === HyperShift Passthrough ===
    // Embeds generated type from api/v1alpha1/zz_generated.passthrough.go
    
    HostedCluster HostedClusterSpecPassthrough `json:"hostedCluster"`
}
```

## How marker-scanner Works

The marker-scanner:
1. Scans **hand-written** files (skips `zz_generated*`)
2. Extracts markers from envelope fields
3. Resolves embedded types via type cache
4. Extracts markers from **all nested fields** in passthrough types

This is why scanning `examples/hypershift/` finds 33 fields even though the `.go` files only define the envelope — the scanner recursively follows the embedded `HostedClusterSpecPassthrough` and `NodePoolSpecPassthrough` types.

## Integration with Generated Code

In a real project, you'd have:

1. **Generated passthrough types** (from passthrough-gen):
   ```
   api/v1alpha1/zz_generated.passthrough.go
   ```
   Contains `HostedClusterSpecPassthrough`, `NodePoolSpecPassthrough`, etc.
   All fields have safe defaults: `+k8s:openapi-gen=false` + `+hyperfleet:write-mode=service-set`

2. **Hand-written envelope types** (like these examples):
   ```
   api/v1alpha1/cluster_types.go
   api/v1alpha1/nodepool_types.go
   ```
   Embed the generated passthrough types and add HyperFleet-specific fields

3. **Field registry** (from marker-scanner):
   ```
   pkg/registry/field_metadata.go
   ```
   Generated by scanning the hand-written envelope types

See `docs/workflow.md` for the complete three-stage pipeline.
